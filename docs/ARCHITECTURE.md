# 🐉 Архитектура HYDRA 2.5.0

Документ описывает текущую модель системы: её слои, границы ответственности и
правила, которые должны сохраняться при эксплуатации и дальнейшем развитии.

## 1. 🧭 Модель системы

HYDRA — оркестратор многопротокольного прокси-сервера. Пользовательские данные
и настройки хранятся в `state.json`, а рабочие конфигурации служб создаются из
него через единый прикладной и оркестрационный слой.

```text
┌────────────────────────────────────────────────────────────┐
│ TUI / командная строка / будущие API-адаптеры              │
└──────────────────────────────┬─────────────────────────────┘
                               │ прикладные службы
┌──────────────────────────────▼─────────────────────────────┐
│ ProtocolService · UserService · ApplicationService          │
└──────────────────────────────┬─────────────────────────────┘
                               │ оркестратор транзакций
┌──────────────────────────────▼─────────────────────────────┐
│ state · Sing-Box · nftables · Caddy L4 · жизненный цикл     │
│ плагинов                                                    │
└──────────────────────────────┬─────────────────────────────┘
                               │ HostBackend
┌──────────────────────────────▼─────────────────────────────┐
│ systemd · файлы · сетевые инструменты · команды хоста       │
└────────────────────────────────────────────────────────────┘
```

Слой представления не должен напрямую управлять `systemd`, межсетевым экраном,
файлами конфигурации или отдельными плагинами. Такие действия проходят через
прикладные службы, оркестратор и `HostBackend`.

## 2. 🌐 Пути входящего трафика

### TCP/443 и SNI

Caddy L4 используется, когда нужен общий TCP/443: включён AnyTLS, TrustTunnel
или Hysteria2 с доменом, одновременно работают несколько TLS-транспортов либо
задан домен сервера подписок.

- `NaiveProxy` и `ShadowTLS` используют сквозную передачу TLS без расшифровки.
- `AnyTLS` завершает TLS в Caddy, затем разделяет протокольный поток и обычный
  HTTP-запрос к сайту-заглушке.
- `TrustTunnel` завершает TLS и отправляет CONNECT во внутренний HTTP-адаптер;
  обычные запросы направляются на сайт-заглушку.
- TCP-маршрут домена Hysteria2 обслуживает сайт-заглушку; сам Hysteria2 работает
  по UDP на своём порту.
- Для неизвестного SNI используется резервный маршрут только при наличии
  соответствующей активной заглушки.

### UDP/443 и QUIC

UDP/443 нельзя распределить по SNI. Прямым владельцем этого порта может быть
только один QUIC-транспорт — NaiveProxy или TrustTunnel; конфликт отклоняется
до применения конфигурации.

Hysteria2 не является владельцем UDP/443: его UDP-порт и TCP-заглушка
управляются отдельно.

### Остальные входы

AmneziaWG и qWDTT создают туннельные интерфейсы, Mieru и Snell работают как
отдельные входящие подключения, а Telemt обслуживает MTProto. Sing-Box применяет
общие правила маршрутизации, DNS и исходящие соединения ко всем потокам,
которые входят в его контур.

## 3. 🔐 TLS-мультиплексор как контролируемый артефакт

Источник истины для доменов — state, а рабочий артефакт —
`/etc/caddy-l4/config.json`. После изменения домена или транспорта выполняется:

```bash
sudo hydra plan
sudo hydra apply
sudo hydra doctor
```

`hydra doctor`, `hydra plan` и `hydra status` содержат блок `tls_mux`. Он
сравнивает ожидаемые и фактические SNI и сообщает:

- `missing` — домен есть в state, но отсутствует в Caddy;
- `stale` — в Caddy остался домен, которого больше нет в state;
- `certificate_errors` — отсутствует сертификат или ключ для завершения TLS;
- `errors` — повреждён JSON-конфигурации или неактивен `caddy-l4`.

Проверка только читает данные. Исправление выполняется через транзакционный
`apply`: HYDRA создаёт временный конфиг, запускает проверку Caddy и откатывает
изменения при неудаче.

## 4. 🔄 Жизненный цикл применения

```text
state.json
   │
   ├─ проверка и миграция
   ├─ сбор фрагментов плагинов
   ├─ создание конфигурации Sing-Box
   ├─ применение nftables / TPROXY
   ├─ применение включённых плагинов
   ├─ перезагрузка Sing-Box
   ├─ пересборка и проверка Caddy L4 (если требуется)
   ├─ управление службой учёта трафика
   ├─ проверка работоспособности
   └─ фиксация или откат
```

Для каждой операции сохраняется снимок. При критическом сбое восстанавливаются
state, Sing-Box, nftables, плагины и связанные рабочие файлы. Откат идёт в
обратном порядке, продолжает работу после локальной ошибки и не выполняется
повторно.

`hydra apply --dry-run` строит план на копии state и не выполняет изменяющих
систему этапов.

## 5. 🧩 Контракты плагинов

Плагин описывает свои возможности, зависимости, фрагмент конфигурации, жизненный
цикл и проверку работоспособности. Реестр проверяет этот контракт до выполнения
команд на хосте.

- Типизированный `ConfigFragment` не допускает произвольные значения Python.
- `configure` подготавливает данные, `apply` применяет их, а `health` подтверждает
  работоспособность.
- Автономные компоненты (например, Honeypot) не перезапускаются общим `apply`,
  если это не требуется их собственным жизненным циклом.
- Старые обработчики с результатом `bool` или `tuple` поддерживаются адаптерами;
  новый код должен возвращать формализованные результаты.

## 6. 💾 State и рабочее состояние

`state.json` содержит только желаемую и долговечную конфигурацию. Факты вроде
`running`, `health` и рассинхронизация вычисляются отдельно через `RuntimeSnapshot`.

Хранилище использует:

- последовательные миграции схемы `vN → vN+1`;
- атомарную замену и синхронизацию каталогов;
- безопасный отказ при неизвестной будущей схеме;
- резервные и изолированные копии при повреждении;
- проверку данных до записи.

Нельзя исправлять фактическое состояние ручной записью в state: сначала нужно
выяснить причину рассинхронизации через `doctor`, `status` или `reconcile`.

## 7. 🛡️ Безопасность и доступ

- Все привилегированные команды проходят через `HostBackend` и имеют таймаут.
- Конфигурации и state записываются атомарно; секреты не выдаются в безопасном
  публичном статусе.
- Backup содержит манифест и SHA-256. Restore проверяет выход за разрешённые
  пути, симлинки и допустимые каталоги, а перед записью создаёт страховочную
  копию.
- `reconcile --apply` исправляет только безопасную рассинхронизацию enable/disable;
  он не устанавливает пакеты и не делает рискованных предположений.

## 8. 📊 Мониторинг и диагностика

Минимальный набор для проверки работающей системы:

```bash
sudo hydra status
sudo hydra doctor
sudo hydra plan
sudo hydra reconcile
journalctl -u sing-box -u caddy-l4 --no-pager -n 100
```

Для автоматизации используйте поля JSON и `error_details`, а не только текст
ошибки. Полное описание команд находится в [CLI.md](CLI.md).

## 9. 🚧 Границы текущей версии

REST API и web-панель не входят в `2.5.0`. Telegram-бот пока не считается
готовым для промышленной эксплуатации: его контракт и проверочные сценарии
должны быть отдельным этапом, чтобы не смешивать экспериментальный транспорт с
проверенными CLI и TUI.

Новый компонент должен подключаться через прикладную службу, описывать контракт
плагина, участвовать в транзакционном жизненном цикле и иметь тесты сценариев
отказа до
того, как его можно считать частью стабильной архитектуры.
