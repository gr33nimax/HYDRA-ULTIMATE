# 🐉 Архитектура HYDRA 2.5.0

Этот документ описывает не историю рефакторинга, а текущую модель системы:
какие слои существуют, кто за что отвечает и какие инварианты должны сохраняться
при эксплуатации и дальнейшем развитии проекта.

## 1. 🧭 Модель системы

HYDRA — это оркестратор многопротокольного прокси-сервера. Состояние пользователя
и настройки хранятся в `state.json`, а runtime-конфигурации генерируются из него
через единый application/orchestration слой.

```text
┌────────────────────────────────────────────────────────────┐
│ TUI / Headless CLI / будущие API-адаптеры                  │
└──────────────────────────────┬─────────────────────────────┘
                               │ application services
┌──────────────────────────────▼─────────────────────────────┐
│ ProtocolService · UserService · ApplicationService          │
└──────────────────────────────┬─────────────────────────────┘
                               │ transactional orchestration
┌──────────────────────────────▼─────────────────────────────┐
│ state · Sing-Box · nftables · Caddy L4 · plugin lifecycle   │
└──────────────────────────────┬─────────────────────────────┘
                               │ HostBackend
┌──────────────────────────────▼─────────────────────────────┐
│ systemd · filesystem · network tools · bounded host commands│
└────────────────────────────────────────────────────────────┘
```

Presentation-слой не должен напрямую управлять `systemd`, firewall, файлами
конфигурации или отдельными плагинами. Все такие действия проходят через
application services, orchestrator и `HostBackend`.

## 2. 🌐 Пути входящего трафика

### TCP/443 и SNI

Caddy L4 используется, когда нужен общий TCP/443: включены AnyTLS, TrustTunnel
или Hysteria2 с доменом, одновременно работают несколько TLS-транспортов либо
задан домен сервера подписок.

- `NaiveProxy` и `ShadowTLS` используют TLS passthrough.
- `AnyTLS` завершает TLS в Caddy, а затем разделяет протокольный поток и HTTP-
  заглушку.
- `TrustTunnel` завершает TLS и отправляет CONNECT-трафик во внутренний
  HTTP-адаптер, обычные запросы идут на decoy.
- TCP-маршрут домена Hysteria2 обслуживает браузерную заглушку; сам Hysteria2
  работает по UDP на своём порту.
- Неизвестный SNI получает fallback только при наличии соответствующей активной
  decoy-схемы.

### UDP/443 и QUIC

UDP/443 не маршрутизируется по SNI. Владельцем raw UDP может быть только один
из поддерживаемых QUIC-транспортов (NaiveProxy или TrustTunnel); конфликт
отклоняется до применения.

Hysteria2 не является владельцем UDP/443-мультиплексора: его UDP-порт и TCP-
заглушка управляются отдельно.

### Остальные входы

AmneziaWG и qWDTT создают туннельные интерфейсы, Mieru и Snell работают как
отдельные inbounds, а Telemt обслуживает MTProto. Sing-Box применяет единые
route rules, DNS и outbounds для тех потоков, которые входят в его контур.

## 3. 🔐 TLS-мультиплексор как контролируемый артефакт

Источник истины для доменов — state, но рабочий артефакт —
`/etc/caddy-l4/config.json`. Поэтому после изменения домена или транспорта
выполняется:

```bash
sudo hydra plan
sudo hydra apply
sudo hydra doctor
```

`hydra doctor`, `hydra plan` и `hydra status` содержат блок `tls_mux`. Он
сверяет ожидаемые и фактические SNI и сообщает:

- `missing` — домен есть в state, но отсутствует в Caddy;
- `stale` — в Caddy остался домен, которого больше нет в state;
- `certificate_errors` — отсутствует cert/key для TLS termination;
- `errors` — повреждённый JSON или неактивный `caddy-l4`.

Аудит read-only. Исправление выполняется только через транзакционный `apply`,
который генерирует pending-конфиг, вызывает Caddy validate и откатывает изменения
при неудаче.

## 4. 🔄 Жизненный цикл применения

```text
state.json
   │
   ├─ validate / migrate
   ├─ collect plugin fragments
   ├─ generate Sing-Box config
   ├─ apply nftables / TPROXY
   ├─ apply enabled plugins
   ├─ reload Sing-Box
   ├─ rebuild + validate Caddy L4 (если нужен)
   ├─ manage traffic daemon
   ├─ health checks
   └─ commit или rollback
```

Каждая операция получает snapshot. При критическом сбое откатываются state,
Sing-Box, nftables, плагины и связанные runtime-артефакты. Rollback выполняется
в обратном порядке, продолжает работу после локальной ошибки и не финализируется
повторно.

`hydra apply --dry-run` строит план на копии state и не проходит mutating-фазы.

## 5. 🧩 Контракты плагинов

Плагин описывает capabilities, зависимости, конфигурационный фрагмент,
lifecycle и health. Registry проверяет контракт до host-операций.

- typed `ConfigFragment` не допускает произвольные Python-значения;
- `configure` подготавливает данные, `apply` применяет их, `health` подтверждает
  рабочее состояние;
- standalone-компоненты (например, Honeypot) не перезапускаются общим apply,
  если это не требуется их собственным lifecycle;
- legacy bool/tuple hooks поддерживаются адаптерами, но новый код должен
  возвращать формализованные результаты.

## 6. 💾 State и runtime

`state.json` содержит только желаемую и долговечную конфигурацию. Факты вроде
`running`, `health` и drift вычисляются отдельно через `RuntimeSnapshot`.

Хранилище использует:

- schema migrations `vN → vN+1`;
- атомарную замену и fsync каталогов;
- fail-closed проверку будущей schema;
- резервные и quarantine-копии при повреждении;
- валидацию перед записью.

Нельзя исправлять runtime-факт ручной записью в state: сначала нужно выяснить
причину drift через `doctor`, `status` или `reconcile`.

## 7. 🛡️ Безопасность и доступ

- Все привилегированные команды проходят через `HostBackend` с таймаутами.
- Конфигурации и state записываются атомарно; секреты не выдаются в безопасном
  публичном статусе.
- Backup содержит manifest и SHA-256, restore проверяет path traversal, симлинки
  и разрешённые каталоги, а перед записью создаёт страховочную копию.
- `reconcile --apply` исправляет только безопасный enable/disable drift; он не
  устанавливает пакеты и не выполняет рискованные догадки.

## 8. 📊 Наблюдаемость

Минимальный эксплуатационный набор:

```bash
sudo hydra status
sudo hydra doctor
sudo hydra plan
sudo hydra reconcile
journalctl -u sing-box -u caddy-l4 --no-pager -n 100
```

Для автоматизации используйте JSON-поля и `error_details`, а не только текст
ошибки. Полное описание CLI находится в [CLI.md](CLI.md).

## 9. 🚧 Границы текущей версии

REST API и web-панель не входят в `2.5.0`. Telegram-бот пока не считается
production-ready адаптером: его контракт и smoke-проверки должны быть отдельным
этапом, чтобы не смешивать экспериментальный transport с проверенным CLI/TUI.

Новый компонент должен подключаться через application service, описывать plugin
contract, участвовать в транзакционном lifecycle и иметь failure-тесты до того,
как его можно считать частью стабильной архитектуры.
