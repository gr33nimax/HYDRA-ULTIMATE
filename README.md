**HYDRA** — это автоматизированная модульная платформа развёртывания, оркестрации и администрирования многопротокольных прокси-серверов на базе **Sing-Box** (используется как единое сетевое ядро). 

Платформа объединяет 14 плагинов (транспорты, сетевые службы, модули безопасности) с централизованной маршрутизацией, DNS-шифрованием, автоматическим распределением портов по SNI с помощью Caddy L4, гибким биллингом и интерактивным TUI-интерфейсом управления.

> [!IMPORTANT]
> Платформа находится в стадии активного бета-тестирования. Рекомендуется развертывание на чистой операционной системе (Ubuntu 20.04+ / Debian 11+).

## 📐 Архитектура системы

HYDRA использует двухуровневую обработку трафика: входящие TLS-соединения мультиплексируются по SNI с помощью **Caddy L4**, а прочие протоколы (например, UDP/TPROXY) обрабатываются непосредственно в ядре **Sing-Box**.

### Схема прохождения трафика

```
                      Входящий TLS-трафик (Порт 443)
                                     │
                                     ▼
      ┌─────────────────────────────────────────────────────────────────────────────┐
      │                   Caddy L4 (SNI-мультиплексор / Mux)                        │
      └──────────────┬───────────────┬───────────────┬───────────────┬──────────────┘
                     │ [sni-1]       │ [sni-2]       │ [sni-3]       │ [unknown/default]
                     ▼               ▼               ▼               ▼
                NaiveProxy        AnyTLS        TrustTunnel     Сайт-заглушка (Decoy)
                 (:10443)        (:20444)        (:20445)       (Защита от сканирования)
                     │               │               │
                     └───────────────┼───────────────┘
                                     │
                                     ▼
  Клиенты (AWG/Mieru/etc.) ──► nftables TPROXY
                                     │
                                     ▼
                             Sing-Box (inbound)
                                     │
                       ┌─────────────┴─────────────┐
                       │                           │
                       ▼ (Политики роутинга)        ▼
                WARP (outbound)            Direct (outbound)
                       │                           │
                       ▼                           ▼
                  Cloudflare                    Интернет
                       │
                 [DNSCrypt:5300] (Локальный шифрованный upstream)

  ──────────────────────────────────────────────────────────────────
  Модули безопасности (Security):
  ┌────────────────────────────────────────────────────────────────┐
  │  Fail2ban  ──► Анализ логов (sing-box/sshd) → Блокировка в nft │
  │  Honeypot  ──► Ловушка сканирования портов → Блокировка        │
  │  IPBan     ──► Статическое управление списками IP/CIDR/ASN     │
  └────────────────────────────────────────────────────────────────┘
```



### Конвейер применения конфигурации (Transaction Lifecycle)

Все изменения настроек в TUI применяются атомарно по транзакционному принципу. В случае ошибки на любом этапе происходит автоматический откат к предыдущему стабильному состоянию системы и базы данных.

```
  TUI (main.py) ────► Запись в AppState (state.json)
                            │
                            ▼ Применение (orchestrator.py)
  1. Сбор фрагментов ───────► registry.collect_fragments()
  2. Генерация конфига ─────► singbox.generate_config()
  3. Валидация изменений ───► Тест структуры JSON Sing-Box & правил nftables
                            │
            ┌───────────────┴───────────────┐
            ▼ Успешно                       ▼ Сбой
  Запись конфигурации на диск        Откат состояния AppState
  Перезапуск служб через Systemd     Восстановление рабочей конфигурации
```

## 🔑 Ключевые механизмы

### 1. Интеграция Cloudflare WARP и выборочная маршрутизация
Маршрутизатор WARP функционирует в режиме **split-tunneling**, направляя в WireGuard-интерфейс только трафик, удовлетворяющий правилам селекции:
* **Мультипрофильные релеи**: Поддержка подключения неограниченного числа сторонних конфигураций WireGuard/AmneziaWG. При размещении файлов конфигурации `.conf` в директории `/etc/hydra/warp_profiles/` (например, `Russia.conf`) система автоматически генерирует для них отдельные исходящие шлюзы (outbounds) и позволяет распределять по ним правила маршрутизации.
* **Автоматическая синхронизация списков**: Фоновая служба `sync_agent.py` каждые 24 часа запрашивает обновленные сигнатуры и списки доменов/IP (включая Geo-block, РФ-ресурсы для сокрытия хоста от сканирования РКН, подсети ИИ-сервисов Google и OpenAI). Перестроение конфигурации Sing-Box и перезапуск сетевого стека происходят бесшовно, без разрыва установленных TCP-сессий.
* **Нативная адресация в Sing-Box (1.13.0+)**: Интеграция осуществляется без промежуточных TUN-интерфейсов на уровне ОС — пакеты направляются напрямую в абстракцию `endpoints` с поддержкой всех расширенных параметров AmneziaWG (CPS-подписи, шумовой трафик, заголовки).

### 2. SNI-мультиплексирование на базе Caddy L4
Для бесконфликтного совместного использования входящего порта TCP/443 несколькими TLS-транспортами HYDRA разворачивает фронтенд-мультиплексор на уровне Caddy с модулем `layer4`:
* **SSL Passthrough (Без дешифрования)**: Маршрутизатор Caddy L4 анализирует структуру пакета `Client Hello` на этапе TLS-рукопожатия, считывает значение поля SNI (Server Name Indication) и перенаправляет сырой зашифрованный поток на соответствующий локальный порт транспорта (например, NaiveProxy на `10443`, AnyTLS на `20444`, TrustTunnel на `20445`). Это исключает раскрытие приватных ключей на уровне мультиплексора.
* **Активная защита от зондирования (Decoy)**: Запросы с некорректным, отсутствующим или неавторизованным SNI автоматически перенаправляются на локальный веб-сервер, отдающий статические decoy-страницы из каталогов `/var/www/decoy-*/`, имитируя работу стандартного веб-ресурса.
* **Спектр UDP/443 (QUIC)**: Для транспортов, поддерживающих протокол QUIC/HTTP3 (NaiveProxy, TrustTunnel), настраивается raw UDP proxy. Оркестратор автоматически отслеживает владение UDP-портом и предотвращает конфликты адресации на этапе валидации изменений.

### 3. Транзакционный биллинг и дистрибуция подписок
Автономный модуль контроля доступа и учета потребляемых ресурсов:
* **HTTPS-сервер подписок**: Развернутый на порту `9443` веб-сервис генерирует конфигурации в реальном времени под целевые клиенты (включая формат NekoBox `sn://`). Блокировка или удаление пользователя приводит к мгновенной аннуляции токена доступа и обрыву активных сессий на сервере подписок.
* **Мониторинг потребления трафика (`traffic_daemon`)**: Фоновая служба опрашивает встроенный Clash API ядра Sing-Box. Полученные метрики агрегируются в локальной БД. Учет расхода ресурсов спроектирован как монотонный процесс: сброс сетевых интерфейсов, перезапуск демонов или ротация системных журналов не вызывают сброса квот или двойного тарифицирования. При исчерпании лимита (`limit_bytes`) или окончании срока действия учетной записи пользователь автоматически деавторизуется во всех inbounds.

### 4. Автоматизированная диагностика VPS
Встроенная интерактивная подсистема (Опция 8 в TUI) позволяет проводить комплексный аудит сетевой доступности и производительности хоста:
* **Проверка блокировок DPI (Censorcheck)**: Сигнатурный анализ доступности ИИ-ассистентов (OpenAI, Gemini), стриминговых платформ (Netflix, YouTube) и детектирование признаков деградации или блокировок трафика техническими средствами противодействия угрозам (ТСПУ).
* **GeoIP & Сетевая телеметрия**: Определение реального географического положения хоста, автономной системы (ASN) и провайдера для верификации маршрутов.
* **RTT и пропускная способность**: Тестирование скорости (Download/Upload) и задержек с помощью iPerf3 до ключевых точек обмена трафиком в РФ и глобального Speedtest до 14 целевых дата-центров по всему миру.
* **Профилирование CPU**: Оценка производительности процессора в однопоточном режиме с помощью `sysbench` для контроля переподписки (overselling) на виртуализации хостинга.

### 5. Совместимость с клиентскими приложениями
При импорте конфигураций через HTTPS-сервер подписок поддерживаются следующие клиенты:

| Клиент | Поддерживаемые транспорты | Совместимость и ограничения |
| :--- | :--- | :--- |
| **NekoBox** | Все доступные протоколы | Полная совместимость. При использовании **TrustTunnel QUIC** имеется баг на стороне клиента — требуется принудительно выбрать режим `quic` в настройках профиля. |
| **Throne** | Все доступные протоколы | Поддерживаются все транспорты, за исключением **TrustTunnel QUIC** (ограничение со стороны клиента Throne). |
| **Karing** | `NaiveProxy`, `AnyTLS` | Полная совместимость с указанными протоколами. |

## 💿 Установка и запуск

### Системные требования
* **ОС**: Ubuntu 20.04+, Debian 11+ (рекомендуется чистая установка)
* **Ресурсы**: ОЗУ от 512 МБ, ПЗУ от 2 ГБ, внешний белый IPv4-адрес
* **Интерпретатор**: Python >= 3.10
* **Права**: `root`

### Быстрая установка (Рекомендуется)
Для автоматической установки сетевого стека, Caddy L4, Sing-Box Extended и создания символических ссылок выполните:
```bash
bash <(curl -fsSL https://raw.githubusercontent.com/gr33nimax/HYDRA-ULTIMATE/main/bootstrap.sh)
```
*Скрипт автоматически установит необходимые зависимости, настроит systemd-службы и скомпилирует caddy-l4 с нужными модулями.*

### Ручная установка
Если вы хотите развернуть проект вручную из исходного кода:
```bash
# Клонирование репозитория в директорию по умолчанию
git clone https://github.com/gr33nimax/HYDRA-ULTIMATE /opt/hydra
cd /opt/hydra

# Запуск панели управления (требуется Python 3.10+)
sudo python3 main.py
```

### Первоначальная настройка
1. Запустите консольное меню: `sudo hydra` (или `sudo python3 main.py`).
2. Перейдите в раздел **Протоколы** и включите необходимые вам транспорты.
3. Активируйте системные службы (**DNSCrypt**, **WARP**).
4. Перейдите в **Пользователи** и создайте аккаунты. Платформа сгенерирует ссылки и QR-коды для подключения.
5. Проверьте статус служб в TUI или с помощью системных команд:
```bash
sudo hydra status
journalctl -u sing-box -n 20 --no-pager
```

## 🧪 Разработка и тестирование

Проект покрыт развитой системой юнит-тестов (более 230 тестов), проверяющих генерацию конфигураций, транзакционный жизненный цикл плагинов, работу Caddy L4 и биллинг-демона.

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/gr33nimax/HYDRA-ULTIMATE.git
   cd HYDRA-ULTIMATE
   ```
2. Установите pytest:
   ```bash
   pip install pytest
   ```
3. Запустите тесты:
   ```bash
   python -m pytest tests/ -v
   ```
   *Все тесты выполняются изолированно в течение ~7 секунд.*

4. Для локальной отладки TUI (требуется Linux c правами `root`):
   ```bash
   sudo python3 main.py
   ```

| Компонент | Описание изменений |
| :--- | :--- |
| **Оркестрация** | Внедрен транзакционный цикл `configure → validate → apply → commit/rollback` с межпроцессными блокировками. |
| **Безопасность** | Исправлено управление портами, удаление правил UFW/iptables, валидация nftables/Caddy перед применением. |
| **Биллинг** | Реализован монотонный учет трафика по Clash API (устойчивый к сбросу интерфейсов и ротации логов). |
| **UI/TUI** | Приведение единиц к IEC (KiB/MiB/GiB), скрытие неактивных пиров AWG. |

### [2.3.4] — 11.07.2026

| Компонент | Описание изменений |
| :--- | :--- |
| **WARP** | Поддержка кастомных WireGuard/AmneziaWG профилей и раздельный роутинг списков. |

### [2.3.3] — 09.07.2026

| Компонент | Описание изменений |
| :--- | :--- |
| **Fail2ban** | Изолированные джейлы для каждого протокола (`recidive`), защита от Portscan для приватных подсетей. |
| **AmneziaWG** | Добавлен мастер настройки обфускации и гейминг-профиль `low_latency`. |
| **Mieru** | Пресеты обфускации в TUI и генерация ссылок `mierus://` для клиента Karing. |

### [2.3.2] — 09.07.2026

| Компонент | Описание изменений |
| :--- | :--- |
| **Мультиплексор** | Бесшовная миграция с HAProxy на Caddy L4. Разрешены конфликты портов NaiveProxy. |
| **Компиляция** | Поддержка кросс-компиляции на ARM64/AMD64. |
