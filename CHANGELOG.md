# Changelog

## [4.11.1] — 2026-05-20

### Исправлено

- **`[CL]` Кластерное управление — SSH-аутентификация по паролю**

  **Симптом:** все операции кластера (`[CL]`) завершались с ошибкой
  `Permission denied (publickey,password)` на всех нодах — диагностика,
  перезапуск, обновление, ротация UUID, произвольная команда и проверка SSH.

  **Причина:** в `_ssh_opts()` был жёстко задан флаг `BatchMode=yes`, который
  явно запрещает любой интерактивный ввод. SSH пробовал ключ, получал отказ
  и сразу падал — без попытки парольной аутентификации.

  **Исправления в `cluster_ops.py`:**
  - SSH-транспорт разделён на два режима: `_ssh_opts_key()` (BatchMode=yes +
    ключ) и `_ssh_opts_pass()` (PreferredAuthentications=password, без BatchMode)
  - Добавлена поддержка `sshpass` для парольной аутентификации; при отсутствии
    пакет автоматически устанавливается через `apt install sshpass`
  - Функция `_ask_password()` — запрашивает пароль root один раз за сессию,
    проверяет его на первой ноде, сохраняет в `_session_password`
  - Функция `_get_creds()` — приоритет: сессионный пароль → ключ (быстрая
    проверка на первой ноде) → запрос пароля
  - Все шесть операций кластера получают `(ssh_key, password)` через `_get_creds()`
    и передают пароль во все вызовы `_ssh()` и `cluster_run()`
  - Новый пункт меню **`[P] Сменить пароль сессии`** — сброс и повторный ввод
    пароля без выхода из меню
  - В заголовке меню кластера отображается текущий режим аутентификации:
    SSH-ключ / пароль (сессия) / нет ключа

- **`[NW]` nginx Watchdog — длинные строки выходили за рамки**

  Строки `_box_item` и `_box_row` с длинным текстом разбиты на 2 строки:
  - «Включить watchdog (timer каждые 2 минуты)» → название + описание отдельной строкой
  - «Watchdog установлен, timer активен (каждые 2 мин)» → 2 строки
  - «Watchdog не установлен — сначала включите его (пункт 1)» → 2 строки

- **`[IP]` ipset Persistent — длинные строки выходили за рамки**

  - «Установить юнит (восстановление при boot)» → название + описание отдельной строкой
  - «Показать содержимое /etc/ipset.conf (первые 30 строк)» → 2 строки
  - «Юнит установлен, ipset восстановится при boot» → 2 строки

- **`[CL]` SSH-проверка — строки с ошибками выходили за рамки**

  SSH-результат разбит на 2 строки: хост на первой, причина ошибки
  (`Permission denied (publickey,password)`) с отступом на второй.
  Дополнительно исправлены длинные строки в других пунктах меню кластера.

## [4.10] — 2025-05-19

### Исправлено

- **[CRITICAL] Отвал клиентов (EOF) после применения RIPE-подсетей при включённой блокировке входящих из РФ**

  **Симптом:** после нажатия «РФ подсети RIPE NCC → Скачать/обновить» при одновременно
  включённой «Блокировке входящих из РФ» все клиенты переставали подключаться с ошибкой
  `outbound/vless[proxy]: EOF`. Помогало только Аварийное восстановление.

  **Причина — deadlock в `_nginx_restart_if_reality`:**
  В режиме REALITY unix-сокет (`/dev/shm/XXXX.socket`) создаётся и принадлежит **nginx**.
  Xray при старте выполнял `ExecStartPre: rm -f /dev/shm/XXXX.socket`, удаляя сокет nginx.
  Функция восстановления ждала появления сокета *до* перезапуска nginx — deadlock,
  потому что сокет появляется только *после* запуска nginx. Цикл ждал 20 секунд,
  не дожидался, делал `return` — nginx не перезапускался, клиенты получали EOF бессрочно.

  **Исправления:**
  - `_nginx_restart_if_reality`: порядок инвертирован — сначала `restart nginx`
    (он создаёт сокет), затем ждём подтверждения появления сокета. Добавлена
    защита для AWG-режима (там unix-сокет не используется).
  - `create_xray_service`: убрана строка `ExecStartPre: rm -f socket` из шаблона
    `xray.service`. Xray не является сервером на этом сокете и не должен его удалять.
  - `_ru_subnets_apply_to_xray`: добавлен патч существующего `xray.service` на лету
    перед каждым рестартом (исправляет установленные системы без переустановки).
    Таймаут ожидания `xray active` увеличен с 15 до 90 секунд
    (конфиг с 13 000+ RIPE-правил поднимается 30–60 сек).
    nginx перезапускается в любом случае даже при истечении таймаута.
  - `_rebuild_and_restart_xray`, `_as_direct_apply_to_xray`: та же логика
    «nginx первым», таймауты 90 сек.
